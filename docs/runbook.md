# Runbook / Операционное руководство

В этом документе собраны основные инструкции по эксплуатации приложения авто-тестнетов.

## Как поднимать проект локально
1. Убедитесь, что установлен Node.js (рекомендуется v20+) и Yarn/npm.
2. Скопируйте файл конфигурации:
   ```bash
   cp .env.example .env
   ```
3. Установите зависимости:
   ```bash
   npm install
   ```
4. Запустите необходимые сервисы (например, Redis, если используется `RedisQueue`). Можно использовать Docker:
   ```bash
   docker-compose up -d redis
   ```
5. Запустите проект в режиме разработки (API + Worker):
   ```bash
   npm run dev
   ```

## Как добавлять новый аккаунт
1. Создайте новую сид-фразу (или приватный ключ) в надежном месте.
2. Подготовьте новый браузерный профиль. Можно использовать встроенный скрипт для инициализации.
3. Добавьте запись об аккаунте в базу данных или конфигурацию.
4. (Рекомендуется) Запустите скрипт `scripts/init-profile.ts` для первоначального импорта сид-фразы в кошелек в визуальном режиме (`PWDEBUG=1`), чтобы избежать капчи или ручных проверок на первом этапе.

## Как создавать новую задачу
1. Перейдите в веб-интерфейс дашборда.
2. Нажмите "Create Run" (Создать задачу).
3. Выберите нужный сценарий (Scenario), профиль браузера (Profile ID) и сеть (Network).
4. Задайте `maxAttempts`, если требуется больше/меньше повторов при сбоях.
5. Нажмите "Submit". Задача появится в статусе `queued`.

## Как запускать задачу вручную
- Обычно задачи подхватываются Worker'ом автоматически из очереди.
- Если нужно запустить конкретный сценарий минуя очередь, используйте CLI утилиту (если реализовано в проекте):
  ```bash
  npm run cli -- run-scenario <scenarioId> <profileId> <networkId>
  ```
- Для повторного запуска `failed` задачи, через API или интерфейс дашборда нужно создать новую задачу (клон) с тем же сценарием и параметрами.

## Как смотреть логи
- По умолчанию логи выводятся в `stdout`. В production используйте `pm2 logs` или настроенную систему сборки логов (ELK, Datadog).
- Каждая лог-запись содержит JSON формат (если используется `pino` или аналог) и тэги `runId`, `attempt`, `scenarioId` для удобной фильтрации.
- Артефакты (скриншоты и HTML дерева страницы) для упавших задач сохраняются локально. Пути до них можно найти в логах или в дашборде (поле `artifacts`).

## Как разбирать ошибки
1. **Найдите ошибку в дашборде или логах**, обратите внимание на `status` (`failed` или `needs_review`) и `code` (например, `wallet_unknown_state`).
2. Проверьте сохраненные скриншоты и HTML. Это самый быстрый способ понять, где скрипт застрял.
3. Если ошибка `transient` (например, таймаут сети или rate limit), скрипт уже должен был попробовать перезапуститься. Проверьте количество `attempt`.
4. В случае `needs_review` или `wallet_unknown_state`, кошелек оказался в непредвиденном интерфейсном состоянии (рекламный баннер, обновление соглашений). Войдите в профиль вручную (см. следующий пункт) и закройте модальные окна.

## Как восстанавливать сломанный browser profile
Если профиль постоянно выдает ошибки или расширение кошелька крашнулось:
1. Завершите все воркеры, чтобы снять локи (или удалите stale `.lock.json` файлы профиля в директории `~/.testnets-locks`).
2. Запустите скрипт восстановления в ручном режиме:
   ```bash
   npm run cli -- open-profile <profileId>
   ```
3. Откроется браузер. Выполните необходимые ручные действия (введите пароль, закройте спам-модалки, переустановите расширение при необходимости).
4. Закройте браузер. После этого можно запускать задачи на этом профиле заново.

## Как обновлять расширения кошельков
1. Скачайте новую версию расширения (crx файл) или обновите папку с распакованным расширением.
2. Поместите новую версию расширения в папку, откуда берутся base extensions для профилей.
3. **Обязательно** локально прогоните тесты (`npm run test:wallet`) с новой версией, чтобы убедиться, что селекторы из `docs/selector-adaptation-guide.md` все еще работают.
4. Если селекторы поменялись, обновите контроллеры кошельков (например, `MetaMaskController`).
5. Задеплойте апдейт. Новые или пересобранные профили будут использовать новую версию.

## Как безопасно ротировать секреты
Секреты (пароли от кошельков) хранятся в зашифрованном виде.
1. Сгенерируйте новый мастер-ключ: `SecretService.generateKey()`.
2. Напишите миграционный скрипт, который:
   - Создает экземпляр `SecretService` со старым ключом (для расшифровки).
   - Создает экземпляр `SecretService` с новым ключом.
   - Читает все пароли из БД/окружения, расшифровывает старым, шифрует новым ключом и сохраняет обратно.
3. Обновите мастер-ключ в переменных окружения (`.env` или секрет-менеджер).
4. Перезапустите приложение.

## Как откатывать неудачные изменения
1. **Код**: Выполните git revert коммита и сделайте redeploy.
2. **Селекторы**: Если новые селекторы не работают на production расширениях, откатите файл контроллера кошелька к предыдущей версии:
   ```bash
   git checkout HEAD~1 src/core/wallet/metamask/MetaMaskController.ts
   ```
3. **База данных/миграции**: Избегайте деструктивных миграций. Если структура RunStore поменялась неудачно, восстановите дамп БД (если используется) или очистите зависшие задачи через БД.
4. **Профили**: Если релиз "повредил" профили браузера (например, скрипт нажал не ту кнопку в кошельке и сбросил настройки), потребуется ручное вмешательство или восстановление их из бэкапа (если директории профилей бэкапируются).
