# New Wallet Adapter Guide

Данное руководство описывает процесс добавления поддержки нового web3 кошелька в проект. Все кошельки в системе абстрагируются через единый интерфейс, что позволяет сценариям работать с любым кошельком (MetaMask, Rabby и т.д.) без изменения бизнес-логики.

## Как добавить новый wallet adapter
1. Создайте новую директорию в `src/core/wallet/<wallet-name>`.
2. Создайте класс-контроллер, например, `PhantomController.ts` или `OkxWalletController.ts`.
3. Реазуйте логику взаимодействия со всплывающими окнами этого кошелька, используя Playwright `PageHandle` или `BrowserContext`.
4. Обновите типы и константы WalletKind, чтобы система знала о новом кошельке.

## Какие интерфейсы реализовать
Ваш новый контроллер должен имплементировать интерфейс `WalletController` (из `src/shared/types.ts`):

```typescript
export type WalletController = {
  kind: WalletKind; // Например, "phantom"
  detect(page: PageHandle, context: WalletContext): Promise<WalletDetection | null>;
  unlock(page: PageHandle, password: string, context: WalletContext): Promise<void>;
  connect(page: PageHandle, context: WalletContext): Promise<void>;
  ensureNetwork(page: PageHandle, request: WalletConnectionRequest, context: WalletContext): Promise<void>;
};
```

- **`detect`**: Определяет, находится ли целевая страница в контексте запроса от этого кошелька.
- **`unlock`**: Ввод пароля для разблокировки кошелька (если он заблокирован).
- **`connect`**: Подтверждение запроса на подключение сайта к кошельку (Approve connection).
- **`ensureNetwork`**: Смена или добавление сети (Switch/Add Network).

## Какие detector rules нужны
Метод `detect` должен возвращать объект `WalletDetection` с уровнем уверенности (`confidence` от 0 до 1).
Признаки для детектирования:
1. **URL страницы**: Если это окно расширения, проверьте `page.url()`. Начинается ли он с протокола расширения (`chrome-extension://`) и совпадает ли ID расширения или путь.
2. **Специфичные DOM элементы**: Наличие уникальных ID или data-атрибутов в HTML (например, `<div id="app-content">` + логотип кошелька).
3. **Глобальные переменные**: В контексте обычных страниц можно проверить `await page.evaluate(() => !!window.phantom)`.

Пример:
```typescript
if (url.includes('chrome-extension://') && url.includes('popup.html')) {
    const hasLogo = await page.locator('.wallet-logo-class').isVisible();
    if (hasLogo) return { kind: WalletKind.Phantom, confidence: 0.9 };
}
```

## Как встроить новый кошелек в factory
(В текущей архитектуре роль factory обычно играет механизм оркестрации кошельков или `WalletManager`, который может принимать на вход массив контроллеров.)
1. Добавьте новый `WalletKind` в `src/shared/types.ts`:
   ```typescript
   export const WalletKind = {
     MetaMask: "metamask",
     Rabby: "rabby",
     Phantom: "phantom" // <---
   } as const;
   ```
2. Если в системе есть место инициализации `WalletManager` (например, в D&I контейнере или при старте Worker/Scenario), прокиньте экземпляр нового контроллера в список доступных для детекции.
3. Убедитесь, что логика мульти-детектирования способна опросить все контроллеры и выбрать тот, у которого `confidence` выше.

## Как протестировать новый adapter
1. **Локальный тест**: Используйте `scripts/local-test.ts` с профилем браузера, в который установлено расширение нового кошелька.
2. **Интерактивный дебаг**: Включите `PWDEBUG=1`. Пройдите флоу вручную, чтобы собрать локаторы для кнопок Confirm, Unlock, Switch Network.
3. **Мокированные тесты**: Сохраните HTML-код страницы кошелька из Playwright (например, страницу ввода пароля) и напишите unit-тест для `detect` и `unlock` методов, "скармливая" им замоканный `PageHandle`.
4. Запустите существующий базовый сценарий `ConnectWalletScenario` с профилем, настроенным на новый кошелек.

## Какие edge cases учитывать
1. **Онбординг (First Time Setup)**: Что делать, если кошелек не инициализирован? Контроллер должен понимать это состояние и сбрасывать задачу в `NeedsReviewError` с кодом `wallet_unknown_state`.
2. **Рекламные баннеры (What's New)**: Некоторые кошельки показывают полноэкранные попапы "Что нового" после обновления. Добавьте проверку и закрытие таких окон перед основным дейстивием.
3. **Параллельные запросы**: Если dApp делает два запроса на подпись одновременно, кошелек может показать очередь.
4. **Окно уже разблокировано**: Метод `unlock` не должен падать, если кошелек уже разблокирован. Проверьте валидность состояния перед действием.

## Как не ломать существующие сценарии
1. **Абстракция**: Сценарии делают вызовы вида `walletManager.connect(page)` или `walletManager.ensureNetwork(...)`. Убедитесь, что ваш адаптер полностью инкапсулирует логику селекторов. Сценарий ничего не должен знать о структуре DOM кошелька.
2. **Соблюдение контракта**: Если адаптер завершает метод (например, `connect()`), сценарий ожидает, что окно кошелька закрылось или перешло в фоновое состояние, а dApp получил подтверждение. Не оставляйте висящие диалоги и явным образом ожидайте закрытия или выполнения действия (`page.waitForURL`, `page.waitForEvent('close')` и т.п., если применимо).
3. **Не трогайте чужие адаптеры**: Старайтесь не создавать общих базовых классов с синглтонами или разделяемым состоянием, которое может повлиять на поведение `MetaMaskController` или `RabbyController`.
